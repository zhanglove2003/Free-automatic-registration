## 1. 总体技术架构

### 1.1 架构目标
- 内置 Chromium 浏览器，承载注册页并执行注入脚本完成自动化。
- 自动化任务与主窗口 UI 解耦，支持并发、可监控、可重试。
- 敏感数据加密落盘；外部集成（接码/邮箱/回调）可插拔扩展。

### 1.2 技术栈推荐（待澄清 Q1 确认）
**推荐：Electron + TypeScript**，理由：
- 交付的 UI 为 HTML/CSS/JS，Electron 可直接复用，无需重写。
- Electron 自带 Chromium，天然满足「内置谷歌浏览器」要求；通过 `BrowserView` / `WebContents` + 预加载脚本 + CDP（Chrome DevTools Protocol）即可实现页面自动化与脚本注入，无需额外 CEF。
- Node.js 生态便于实现 HTTP（接码/邮箱 Worker/回调/打码）、SQLite、加密、调度。

> 备选：①CEF + C#/.NET（CefSharp）；②Tauri + 侧载 Chromium。若需求方指定，则按 Q1 调整本章及后续实现，但模块划分与数据流保持不变。

下文以 **Electron + TypeScript** 为基准描述。

### 1.3 内置浏览器自动化方案
- **承载方式**：每个并发注册任务使用独立的 `BrowserView`（或独立 `BrowserWindow`，离屏/隐藏），绑定独立 `session.partition`（如 `persist:task-<id>` 或内存分区），实现 cookie/缓存隔离。
- **脚本注入**：通过 `webPreferences.preload` 预加载脚本注入自动化运行时；页面级操作（填表、点击、读 DOM）由注入脚本 + `webContents.executeJavaScript` 执行。
- **精细控制**：通过 CDP（`webContents.debugger.attach`）实现网络拦截、请求头改写、Cookie 读取、等待网络空闲、模拟输入等高可靠操作。
- **资源隔离**：每任务可配置代理（`session.setProxy`）与 UA/指纹（待澄清 Q6）。

### 1.4 分层架构
```

└───────────────▲───────────────────────────┬───────────────┘

    │           │           │           │           │
```

## 2. 模块划分

> 每个模块给出：职责、对外接口（内部 API）、依赖、关键设计点。模块间仅通过定义好的接口交互，便于替换与测试。

### 2.1 浏览器控制模块（BrowserController）
- **职责**：创建/销毁浏览器会话、导航、超时与网络空闲等待、截图快照、Cookie/Storage 读取、代理与 UA 设置、CDP 附加。
- **接口**：
  - `createSession(taskId, opts): SessionHandle`
  - `navigate(handle, url): void`
  - `waitFor(handle, condition): void`（选择器出现/网络空闲/URL 变化）
  - `snapshot(handle): screenshotPath`
  - `getCookies(handle) / getStorage(handle)`
  - `destroy(handle)`
- **关键点**：会话隔离（独立 partition）；统一超时与异常封装；所有底层 Chromium 操作收口于此，上层不直接碰 Electron API。

### 2.2 脚本/注入模块（ScriptInjector / SiteAdapter）
- **职责**：按「站点适配描述」执行填表、点击、提交、读取结果；将站点差异封装为**站点适配器（SiteAdapter）**。
- **站点适配器接口**（每站点实现一份）：
  - `fillForm(ctx, profile)`：填邮箱/用户名/密码/手机号
  - `submit(ctx)`
  - `needsCaptcha(ctx): type|null`
  - `needsSms(ctx): bool` / `fillSms(ctx, code)`
  - `needsEmail(ctx): bool` / `applyEmail(ctx, codeOrLink)`
  - `verifySuccess(ctx): bool`（成功判定）
  - `harvest(ctx): {cookies, token, extra}`（抓取登录态）
- **关键点**：适配器以**配置驱动 + 可选自定义代码**结合（选择器走配置；复杂逻辑走代码钩子）。新增站点 = 新增一个适配器，不改核心。

### 2.3 接码模块（SmsProvider，smshero/HeroSMS 实现）
- **职责**：申请号码、轮询取码、释放/标记号码；封装为 `SmsProvider` 接口，HeroSMS（SMS-Activate 协议）为其一种实现。
- **接口**：`acquireNumber(opts) / pollCode(orderId, timeout) / release(orderId) / markInvalid(orderId)`
- **选号策略（已确认 R1 补充）**：支持配置**三个候选国家**，并提供两种选号模式——
  - `country_first`（国家优先）：按配置的国家优先级依次尝试取号；
  - `price_first`（低价优先）：在三个候选国家中选当前最低价的可用号码。
  - 选号策略、候选国家列表、OpenAI 服务码均为配置项（详见《03 集成对接契约》第 2 章）。
- **关键点**：可配置超时/间隔/最大轮询；遵循 HeroSMS（handler_api.php）协议；接口抽象便于未来接其他接码平台。

### 2.4 邮箱监听模块（EmailReceiver，Cloudflare Email Worker 实现）
- **职责**：生成 Catch-all 邮箱地址、通过 Worker HTTP 拉取接口按收件人定位邮件、解析验证码/激活链接（**不使用 IMAP**）。
- **接口**：
  - `allocAddress(): emailAddress`（在 Catch-all 域下生成随机本地名）
  - `waitMail(toAddress, matcher, timeout): MailMessage`（轮询 Worker `GET /mail?to=...`）
  - `extractCode(mail, rule): string` / `extractLink(mail, rule): url`
- **关键点**：见第 7 章邮箱实现细节；解析规则按站点可配置。

### 2.4b 打码模块（CaptchaSolver，已确认 R1）
- **职责**：求解 OpenAI 注册页的 **Cloudflare Turnstile / Arkose FunCaptcha**，保持全自动。
- **接口**：`solve(task: CaptchaTask): Promise<{ token: string }>`（task 含类型、站点 key、页面 URL、必要的 blob/数据）。
- **关键点**：实现为可替换适配器，支持 CapSolver / YesCaptcha / 2Captcha 等；平台与密钥**部署时由用户配置**，不硬编码；统一超时/重试/计费日志；密钥入保险箱、不入日志。

### 2.5 数据处理与回调模块（AccountStore + CallbackDispatcher）
- **职责**：组装账号记录、加密落库、触发可扩展回调推送、维护回调重试队列。
- **接口**：
  - `save(accountRecord): id`
  - `dispatch(record, targets[])`：按适配器推送
  - `CallbackAdapter` 接口：`send(record): result`（HTTP Webhook / 文件 / DB / 平台 API）
- **关键点**：回调适配器注册表（见第 9 章扩展点）；幂等键 + 重试队列 + 失败告警。

### 2.6 日志与监控模块（Logger + Telemetry）
- **职责**：分步结构化日志、任务级日志聚合、向 UI 推送统计与活动流。
- **接口**：`log(taskId, step, level, msg, meta)` / `getTaskLog(taskId)` / `metricsSnapshot()`
- **关键点**：日志分级（debug/info/warn/error）；敏感字段脱敏后再落日志；统计快照供仪表盘订阅。

### 2.7 导入导出模块（ImportExport）
- **职责**：按字段规范导出 CSV/JSON；（可选）导入待注册资料池。
- **接口**：`exportAccounts(filter, format)` / `importProfiles(file)`（待澄清 Q15）
- **关键点**：导出敏感字段需显式开关并二次确认（对应 PRD AC-12）。

### 2.8 调度与编排（Scheduler / Orchestrator，主进程核心）
- **职责**：任务队列、并发闸门（信号量）、任务状态机驱动、资源生命周期编排（号码/邮箱/会话）、重试调度。
- **接口**：`enqueue(taskSpec) / start() / pause() / retry(taskId) / status()`
- **关键点**：见第 3 章进程线程模型与第 8 章容错。

## 3. 进程与线程模型

### 3.1 进程模型（Electron 基准）
- **主进程（Main）**：唯一编排者。承载调度器、状态机、配置中心、凭据保险箱、数据库、日志中心、外部服务客户端（smshero / 邮箱 Worker / 打码 / 回调）。所有「决策」集中于此。
- **渲染进程（Renderer，主窗口）**：仅渲染交付的面板 UI，通过 IPC 上报事件、订阅状态，不直接执行业务与外部调用。
- **任务浏览器进程**：每个并发注册任务对应一个 `BrowserView`/隐藏 `BrowserWindow`，其渲染进程承载注册页与注入脚本，独立 session 分区，进程级隔离（崩溃不影响主窗口与其他任务）。

### 3.2 并发与"线程"
- Node 主线程为单线程事件循环；并发注册靠**异步 I/O + 多浏览器会话**实现，而非多线程。
- **并发闸门**：用信号量限制同时运行的任务数（最大并发，配置项，Q10）。
- **CPU 密集型工作**（如大批量加解密、导出、复杂解析）放入 `worker_threads` 或 `utilityProcess`，避免阻塞主线程编排与 IPC。
- **轮询类等待**（短信/邮件）为定时异步任务，不独占线程；多个任务的轮询共享事件循环。

### 3.3 任务生命周期与资源编排
```
enqueue → [信号量获取] → running
  ├─ 分配 SMS 号码 / 邮箱地址（失败→重试/失败）
  ├─ 创建浏览器会话（独立分区/代理）
  ├─ 执行 SiteAdapter 步骤（填表→提交→校验）
  ├─ wait_sms / wait_email / wait_captcha（暂态，异步轮询）
  ├─ harvest 登录态 → 组装记录
  ├─ AccountStore.save（加密落库）
  ├─ CallbackDispatcher.dispatch（异步重试队列）
  └─ 释放资源（号码标记、销毁会话）→ [信号量释放] → success/failed
```

### 3.4 IPC 契约（Renderer ↔ Main）
| 方向 | 通道 | 载荷 |
|------|------|------|
| R→M | `cmd:start` | 运行配置（站点、数量、并发） |
| R→M | `cmd:retry` | `{taskId}` |
| R→M | `cmd:getTaskLog` | `{taskId}` |
| R→M | `cmd:saveSettings` | 凭据与参数（写入保险箱） |
| R→M | `cmd:export` | `{filter, format}` |
| M→R | `evt:stats` | 统计快照（运行中/排队/完成/失败） |
| M→R | `evt:taskUpdate` | 单任务状态变更 |
| M→R | `evt:activity` | 活动日志条目 |

> IPC 使用 `contextBridge` 暴露白名单 API，渲染层禁用 `nodeIntegration`，开启 `contextIsolation`（安全基线）。

## 4. 数据存储方案

### 4.1 存储选型
- **本地数据库：SQLite**（通过 `better-sqlite3` 或等价库）。单机、零依赖、事务可靠。
- **加密策略（二选一，推荐 A）**：
  - **A. SQLCipher 全库加密**：整库透明加密，密钥经 Windows DPAPI / Electron `safeStorage` 保护。
  - **B. 字段级加密**：库不加密，敏感字段（password/cookies/token/proxy）以 AES-256-GCM 单独加密，密钥同样经 `safeStorage` 保护。
- **密钥管理**：主密钥不硬编码、不明文落盘；用 Electron `safeStorage`（底层 Windows DPAPI，绑定当前用户）加密存储。每次启动解密载入内存。

### 4.2 数据表（建议）
| 表 | 关键字段 | 说明 |
|----|----------|------|
| `accounts` | PRD 第 7 章全字段 | 账号记录，敏感字段密文 |
| `tasks` | id, site, status, attempts, last_error, created_at, finished_at | 任务与状态机 |
| `jobs` | id, params, created_at | 一次批量运行（与 task 1:N，Q11 确认层级） |
| `settings` | key, value(加密) | 凭据与参数（保险箱） |
| `logs` | id, task_id, step, level, msg, meta, ts | 分步日志 |
| `callbacks` | id, account_id, target, status, attempts, next_retry_at | 回调重试队列 |

### 4.3 数据完整性
- 关键写入用事务；账号落库成功后再触发回调（避免回调成功但本地丢失）。
- 异常退出恢复：启动时扫描 `running` 残留任务→按策略标记 `failed/可重试`；`callbacks.pending` 重新投递（对应 PRD AC-13）。

## 5. 接口设计

### 5.1 内部模块接口（汇总）
> 所有内部接口以 TypeScript 接口定义，主进程内通过依赖注入装配；模块间不共享可变全局状态。

```ts
interface SmsProvider {
  // AcquireOpts 含国家候选列表 + 选号策略（country-first / price-first）
  acquireNumber(opts: AcquireOpts): Promise<{ orderId: string; phone: string; country: string }>;
  pollCode(orderId: string, timeoutMs: number): Promise<string>;
  release(orderId: string): Promise<void>;
  markInvalid(orderId: string): Promise<void>;
}

// 打码/人机验证求解器（R1：对接第三方打码平台，可替换）
interface CaptchaSolver {
  readonly name: string;                 // capsolver / yescaptcha / 2captcha ...
  // 支持 Arkose FunCaptcha 与 Cloudflare Turnstile
  solve(req: {
    type: 'funcaptcha' | 'turnstile';
    siteKey: string; pageUrl: string; extra?: object;
  }, timeoutMs: number): Promise<{ token: string }>;
}

interface EmailReceiver {
  allocAddress(): Promise<string>;
  // Cloudflare Email Worker 拉取/接收：按收件人地址精确定位
  waitMail(to: string, matcher: MailMatcher, timeoutMs: number): Promise<MailMessage>;
  extractCode(mail: MailMessage, rule: ExtractRule): string | null;
  extractLink(mail: MailMessage, rule: ExtractRule): string | null;
}

interface SiteAdapter {
  readonly site: string;
  fillForm(ctx: PageCtx, profile: Profile): Promise<void>;
  submit(ctx: PageCtx): Promise<void>;
  needsCaptcha(ctx: PageCtx): Promise<CaptchaType | null>;
  needsSms(ctx: PageCtx): Promise<boolean>;
  fillSms(ctx: PageCtx, code: string): Promise<void>;
  needsEmail(ctx: PageCtx): Promise<boolean>;
  applyEmail(ctx: PageCtx, codeOrLink: string): Promise<void>;
  verifySuccess(ctx: PageCtx): Promise<boolean>;
  // ChatGPT 适配器须在此完成 OAuth 登录并抓取 token 包
  harvest(ctx: PageCtx): Promise<{
    cookies: object; token?: string;
    oauth?: { accessToken: string; idToken?: string; refreshToken?: string;
              chatgptAccountId?: string; workspaceId?: string };
    extra?: object;
  }>;
}

interface CallbackAdapter {
  readonly name: string;
  send(record: AccountRecord): Promise<{ ok: boolean; detail?: string }>;
}
```

### 5.2 外部服务接口约定
> 端点/字段以各平台**官方文档**为准；下表为接入约定，未知项见 PRD 待澄清。

**smshero（HTTP API，SMS-Activate 兼容协议）**
- HeroSMS 为 SMS-Activate 继任平台，沿用其 `handler_api.php` 协议（已核实）。
- 鉴权：`api_key` 作为查询参数。
- 典型调用：`action=getNumber`(申请号码→`ACCESS_NUMBER:id:phone`) → `action=getStatus`(轮询→`STATUS_OK:code`) → `action=setStatus`(完成/取消/重取)。
- OpenAI/ChatGPT 服务码按平台映射配置（SMS-Activate 体系常用 `dr`/`ai`，以 hero-sms 实际为准）。
- 客户端封装：统一超时、重试、状态码映射；Key 来自保险箱，不入日志。详见《03 集成对接契约》。

**Cloudflare Email Routing（见第 7 章，已确认 Q9）**
- 接收侧：Catch-all → 绑定 **Email Worker** 处理来邮；读取侧：Worker 写入 KV/D1 + HTTP 拉取接口（**不使用 IMAP**）。
- 接口端点与鉴权见《03 集成对接契约》邮箱章。

**回调（HTTP Webhook 默认适配器）**
- 方法：`POST <配置URL>`，`Content-Type: application/json`。
- 头：`X-Signature: hmac-sha256(body, secret)`、`X-Idempotency-Key: <account.id>`。
- 体：账号记录 JSON（按字段开关决定是否含敏感字段，默认脱敏）。
- 期望响应：2xx 视为成功；非 2xx 或超时进入重试队列。

### 5.3 回调载荷示例
```json
{
  "id": "uuid",
  "site": "example",
  "username": "user_x9f2",
  "email": "u8f3k2@yourdomain.com",
  "phone": "+1xxxxxxx",
  "account_status": "login_ok",
  "task_id": "task-uuid",
  "created_at": "2026-06-30T08:00:00Z",
  "secrets": { "password": "***", "token": "***" }
}
```
> `secrets` 字段是否明文/脱敏由下游契约与导出策略决定（Q3/Q4 + 安全章）。

## 6. 安全设计

### 6.1 凭据与敏感数据存储
- 所有外部凭据（smshero Key、邮箱 Worker Token、打码平台 Key、回调 secret）与账号敏感字段（password/cookies/token/proxy）**禁止明文落盘**。
- 主密钥经 Electron `safeStorage`（Windows DPAPI，绑定当前用户账户）保护；应用层用 AES-256-GCM 加解密（或 SQLCipher 整库加密）。
- 内存中敏感值用后尽快清理，不写入崩溃转储；日志统一脱敏（掩码中间字符）。

### 6.2 传输加密
- smshero / 回调：强制 HTTPS，校验证书。
- 邮箱：Catch-all → Email Worker → KV/D1 + HTTP 拉取；强制 HTTPS，拉取接口带访问密钥/签名。
- 回调请求带 HMAC 签名，防篡改；幂等键防重复入库。

### 6.3 应用安全基线（Electron）
- `contextIsolation: true`、`nodeIntegration: false`、`sandbox: true`（任务渲染进程）。
- 预加载脚本通过 `contextBridge` 仅暴露白名单 API。
- 注册页为不可信外部内容，严格隔离于主窗口会话与 Node 能力之外。
- 任务会话使用独立 `session.partition`，防止 Cookie/Storage 跨任务串号。

### 6.4 最小权限与审计
- 配置项区分「读」「写」；导出敏感字段需显式开关 + 二次确认并审计记录（AC-12）。
- 日志保留敏感操作审计项（谁/何时/导出了什么），但不含明文密钥。

## 7. 邮箱接收实现细节（Cloudflare Email Routing + Email Workers）
> 已确认（Q9）：采用 **Cloudflare Email Routing + Email Workers/API**，**不使用 IMAP**。完整部署与接口见《03 集成对接契约》邮箱章；本章为设计要点。

### 7.1 接收链路
1. 在 Cloudflare 为域名启用 **Email Routing**，并将 **Catch-all** 规则的动作设为「发送到 **Email Worker**」（而非转发到第三方收件箱）。
2. 注册时为每个任务生成唯一本地名地址（如 `<random>@yourdomain.com`）；该域全部来信进入同一个 Email Worker 的 `email()` 入口。

### 7.2 Worker 处理与存储
- Email Worker 在 `email(message, env, ctx)` 中读取 `message.to`（收件人）、`message.from`、`message.headers`、原始 MIME（`message.raw`），用 MIME 解析库提取 text/html 正文。
- Worker 将结构化结果写入 **Workers KV**（键含收件人地址 + 时间戳）或 **D1**：`{ to, from, subject, text, html, code_candidates[], links[], received_at }`，设置 TTL（如 10 分钟）自动清理。
- 暴露受保护的 **HTTP 拉取接口**（Worker 路由，Bearer Token 鉴权）供注册机查询：`GET /mail?to=<addr>&since=<ts>`；或由 Worker 在收到目标域邮件时**主动 POST 回调**到注册机本地端点（二选一，详见《03》）。

### 7.3 定位与拉取
- 因 Catch-all 汇入同一 Worker，**必须按收件人地址精确定位**：以 `message.to` 为 KV/D1 主键维度，注册机按本任务分配地址查询。
- 拉取策略：可配置间隔（如 5s）、超时（如 120s）、最大次数；命中后删除/标记该 KV 项，避免重复匹配。

### 7.4 验证码 / 链接解析
- 同时解析纯文本与 HTML（HTML 先转文本再匹配）。解析可在 Worker 内预提取 `code_candidates`/`links`，客户端再按站点规则二次确认。
- **验证码**：按站点配置正则（如 `\b(\d{4,8})\b` 配合「verification code / 验证码」上下文约束），降低误匹配。
- **激活链接**：按站点配置 URL 特征（域名/路径前缀）提取，交由 BrowserController 在该任务会话中访问完成验证。
- 解析规则随「站点适配描述」维护，支持每站点覆盖。

### 7.5 失败与边界
- 超时未到邮件：触发重试策略（可重发/换地址，见 Q8）。
- 多封匹配：取最新一封；解析失败记录原文引用（脱敏）便于排查。
- Worker 与 KV/D1 的配额、鉴权 Token 轮换在《03 集成对接契约》中约束。

## 8. 异常处理与容错策略

### 8.1 分层重试
- **步骤级**：单步（取码、收邮件、点击）失败按**指数退避 + 抖动**重试，次数可配。
- **任务级**：步骤级耗尽后整任务重试；超过任务最大重试 → `failed`，记录 `last_error` 与失败步骤。
- **回调级**：独立重试队列，按 `next_retry_at` 退避重投；多次失败转「待重投」并告警。

### 8.2 超时与资源回收
- 每步与整任务均有超时上限；超时即判失败并**确保释放资源**（标记/释放 smshero 号码、销毁浏览器会话、释放并发信号量），防止资源泄漏与号码计费浪费。

### 8.3 幂等与防重
- 账号记录以 `id` 为幂等键；回调带幂等键，下游可去重。
- 落库成功后再回调；回调失败不影响本地已成功数据（AC-13）。

### 8.4 崩溃恢复
- 启动自检：扫描 `tasks.status=running` 残留 → 标记 `failed`(可重试)；重投 `callbacks` 中 `pending/failed` 且未超限项。
- 数据库写入事务化，避免半成品记录。

### 8.5 错误分类
| 类别 | 例 | 处理 |
|------|----|------|
| 可重试瞬时错误 | 网络超时、取码暂未到 | 退避重试 |
| 资源类错误 | 无可用号码、邮件 Worker/KV 不可达 | 延迟重试/告警，必要时暂停作业 |
| 业务终止错误 | 站点拒绝、账号已存在、强人机验证无法通过 | 标记失败，不无限重试 |
| 配置类错误 | 凭据无效、选择器失效 | 立即告警，阻断该作业并提示修正 |

## 9. 集成扩展点设计（CPA / codex manager）

### 9.1 适配器注册表
- `CallbackAdapter` 与导出 `Exporter` 均通过**注册表 + 工厂**装配；新增下游 = 实现接口 + 注册，无需改核心。

### 9.2 字段映射层
- 在适配器内置**字段映射（mapping）**：将内部 `AccountRecord` 映射为下游所需结构（codex manager / CPA 各自的字段名与格式，Q3/Q4 提供后填充）。
- 映射以配置（JSON/模板）驱动，减少为每平台写死代码。

### 9.3 codex manager 集成（已确认 Codex-Manager）
- 下游为开源 **Codex-Manager**（Tauri+Rust），以 **JSON token 包**导入账号（源码核实见《03 集成对接契约》）。
- 实现 `CodexManagerExporter`：将 `AccountRecord` 的 `oauth_*` 字段写为兼容 JSON（`access_token` 必填，`id_token`/`refresh_token`/`chatgpt_account_id`/`email`/`meta` 可选），落盘到约定目录供其「目录导入」，或调用其 `account/import` 入口。

### 9.4 CPA 平台集成预留
- 预留 `CpaExporter` / `CpaCallbackAdapter`：支持「一键导入」所需的文件格式或 API（Q4）。
- 支持多目标并行推送与按目标过滤账号。

### 9.5 接码 / 邮箱 / 打码可替换

## 10. 配置与可观测性

### 10.1 配置项（存于加密 settings）
- smshero：apiKey（部署时由用户输入）、baseUrl、服务码（默认 dr）、**国家候选列表 + 选号策略（country_first / price_first）**、operator、超时、轮询间隔、最大轮询次数。
