## 2. 接码平台契约：HeroSMS（SMS-Activate 协议）

### 2.1 协议性质
HeroSMS 为 SMS-Activate 继任平台，沿用其 HTTP API 协议。所有调用形如：
```
GET  https://<hero-sms-host>/stubs/handler_api.php?api_key=<KEY>&action=<ACTION>&...
```
> 实际 host 与路径以需求方提供的 `https://hero-sms.com/cn/api` 文档为准；下列 action 为 SMS-Activate 标准集，开发时以官方文档逐一校对。

### 2.2 核心调用序列（注册任务内）
| 步骤 | action | 关键参数 | 返回（成功） | 说明 |
|------|--------|----------|--------------|------|
| 查余额 | `getBalance` | `api_key` | `ACCESS_BALANCE:<amount>` | 启动前校验 |
| 申请号码 | `getNumber` | `service`, `country`, `operator?` | `ACCESS_NUMBER:<id>:<phone>` | 取得订单 id + 手机号 |
| 轮询验证码 | `getStatus` | `id` | `STATUS_OK:<code>` / `STATUS_WAIT_CODE` | 轮询至取码或超时 |
| 设置状态 | `setStatus` | `id`, `status` | `ACCESS_*` | status=1 短信已发/8 取消/3 重发/6 完成 |

### 2.3 OpenAI 服务码与国家选择策略（已确认 R1/R3）
- SMS-Activate 体系下 OpenAI（ChatGPT）服务码常用 **`dr`**；HeroSMS 须以其官方服务列表为准。
- **已确认需求**：
  - 支持**最多三个候选国家**，用户在设置中配置（含优先级顺序）。
  - 提供**选号策略开关**：
    - `country_first`（国家优先）：按用户设定的国家优先级依次尝试 `getNumber`，前一国无号再降级到下一国；
    - `price_first`（低价优先）：在三个候选国家中比较实时价格，取最低价国家申请。
  - 目的：用优质/合适国家的手机号注册以**提高账号权重**。
- **实现要求**：服务码、三国候选与优先级、选号策略、operator 全部为**配置项**，不得硬编码。
- 价格查询：用 SMS-Activate `getPrices`（action）获取各国价格供 `price_first` 比较（以官方文档为准）。

### 2.4 失败/异常语义
| 返回 | 含义 | 处理 |
|------|------|------|
| `NO_NUMBERS` | 无可用号码 | 退避重试或换国家码 |
| `NO_BALANCE` | 余额不足 | 中止并告警 |
| `STATUS_WAIT_CODE` | 码未到 | 继续轮询（受最大次数/超时限制） |
| `STATUS_CANCEL` | 订单取消 | 换号重试 |
| `BAD_KEY` / `ERROR_SQL` | 鉴权/平台错误 | 中止并告警 |

### 2.5 配置项
`apiKey`（部署时由用户输入）、`baseUrl`、`service`（默认 dr）、`countries[]`（最多 3 个，含优先级）、`selectionStrategy`（country_first / price_first）、`operator`、`pollIntervalSec`、`pollMaxAttempts`、`acquireRetry`、取码总超时。

## 3. 目标网站契约：ChatGPT / OpenAI 注册

### 3.1 注册与登录本质（决定数据产出）
- OpenAI 账号体系基于 **OAuth（auth0：`auth.openai.com`）**，Codex CLI/Codex-Manager 需要的是登录后的 **OAuth token 包**（access_token / id_token / refresh_token），而非仅用户名密码。
- **因此注册流程不止"注册成功"，必须走完一次 OAuth 授权登录，抓取 token**，否则产出的账号无法纳入 Codex-Manager（见第 4 章字段契约）。

### 3.2 标准流程（站点适配器实现）
```
打开 chatgpt.com / auth.openai.com 注册入口
 → 填邮箱（Cloudflare Catch-all 地址）
 → 邮箱验证（Email Workers 取验证码/链接，见第 5 章）
 → 设置密码
 → 填写出生日期/姓名等资料
 → 人机验证（见 3.3）
 → 手机号验证（HeroSMS 取码，见第 2 章）
 → 注册完成
 → 触发 OAuth 登录授权，抓取 access_token/id_token/refresh_token + chatgpt_account_id
 → 组装账号记录 → 落库 → 回调/导出（第 4 章）
```

### 3.3 风控与人机验证（已确认：对接打码平台）
- OpenAI 注册具备**强人机风控**：常见 **Cloudflare Turnstile** 与 **Arkose Labs FunCaptcha**，且对 IP/指纹敏感。
- **已确认策略（R1）：对接第三方打码/验证码求解平台**，保持全自动。
  - 平台需支持 **Arkose FunCaptcha** 与 **Cloudflare Turnstile**（如 CapSolver / YesCaptcha / 2Captcha 等，最终平台与密钥**部署时由用户配置**，不硬编码）。
  - 实现为可替换的 `CaptchaSolver` 接口（见《02 设计规范》），便于切换打码平台。
- **已确认（R2）：本期不使用代理 IP。**
  - ⚠️ **风险提示（已知并接受）**：OpenAI 对 IP 极敏感，无代理 + 自动化浏览器会显著抬高 Turnstile/FunCaptcha 触发频率，并可能导致 IP 段被风控；这将**直接拉低注册成功率、抬高打码调用成本**。后续如成功率不达预期，首选补救措施即引入住宅代理（接口已预留 FR-15）。
- **手机号提权重策略（R1 补充，归入接码）**：通过选择优质国家手机号注册以提高账号权重，详见第 2 章接码国家策略。
- **成功率与单任务耗时阈值（R7）**：需求方本期暂不设硬性要求，上线后据实统计。

### 3.4 OAuth token 抓取实现建议
- 用 CDP（`webContents.debugger`）拦截 OAuth 回调与 token 端点响应，或读取登录后客户端持有的 token（auth0 回调 / 本地存储 / 网络响应体）。
- 目标产出与 Codex CLI `auth.json` 一致的三件套 + `chatgpt_account_id`（见 4.2）。

## 4. 下游契约：Codex-Manager 账号导入（源码核验）

### 4.1 导入通道（三选一，软件回调优先用 A 或 C）
| 通道 | 机制 | 适用 |
|------|------|------|
| A. 目录导入 | Codex-Manager「选择账号导入目录」递归扫描 `*.json` 批量导入（`import.rs: read_account_import_contents_from_directory`） | 本软件按账号写出 JSON 到约定目录，最稳妥 |
| B. 文件导入 | 选择 `.json`/`.txt` 文件导入 | 手动/小批量 |
| C. RPC 导入 | `account/import`，参数 `{ "contents": ["<jsonString>", ...] }`（`account_import.rs`） | 若 Codex-Manager 本地网关可达，可直接推送 |

> 推荐主路径 **A（写 JSON 到导入目录）**：与本软件「文件/回调」适配器天然契合，且无需依赖 Codex-Manager 运行时端口。C 作为可选直连适配器。

### 4.2 账号 JSON 字段契约（强约束，源自 `extract_token_payload` / `extract_account_meta`）
单账号一个 JSON 对象（也支持顶层数组或 JSON 流）。字段：

```jsonc
{
  // —— token 三件套（核心，access_token 必填）——
  "access_token": "<OAuth access_token>",      // 必填；亦可放入 "tokens": { ... }
  "id_token": "<OAuth id_token>",              // 可选但强烈建议（含 chatgpt_account_id/workspace 等 claims）
  "refresh_token": "<OAuth refresh_token>",    // 可选但强烈建议（用于续期）
  "account_id": "<可选, 账号 id 提示>",
  "chatgpt_account_id": "<可选, 显式 chatgpt 账号 id>",

  // —— 顶层辅助 ——
  "email": "<注册邮箱>",                        // 用作账号 label 兜底

  // —— meta（可选，便于管理）——
  "meta": {
    "label": "<显示名, 默认取 email>",
    "issuer": "<默认 https://auth.openai.com>",
    "group_name": "<分组名>",
    "note": "<备注>",
    "tags": "<标签, 逗号分隔或数组>",
    "workspace_id": "<工作区 id>",
    "chatgpt_account_id": "<同上, meta 内亦可>"
  }
}
```

**关键规则（来自源码）**：
- `access_token` 缺失 → 该条导入失败（`required_string_any`）。
- 字段同时支持 snake_case 与 camelCase（`access_token`/`accessToken` 等）。
- token 字段可平铺在顶层，或包在 `tokens` 对象内（`item.get("tokens").unwrap_or(item)`）。
- `chatgpt_account_id` / `workspace_id` 若不显式给出，Codex-Manager 会尝试从 `id_token`/`access_token` 的 JWT claims 解析 → **务必提供有效 `id_token`**。
- 去重/更新键：Codex-Manager 以 chatgpt_account_id / workspace / subject / refresh_token 指纹判定 upsert，**同一账号重复导入会 update 而非重复创建**（回调可安全重投，天然幂等）。

### 4.3 本软件输出实现
- 账号成功后，由「回调/导出模块」按 4.2 schema 生成 JSON：
  - 文件名建议 `<email>_<chatgpt_account_id|task_id>.json`，写入配置的「Codex-Manager 导入目录」。
  - 或经 C 通道 `account/import` 推送 `{contents:[jsonString]}`。
- 敏感 token 在本软件**本地库内加密存储**；导出为明文 JSON 供导入时，须在 UI 明确提示并经确认（对应 PRD AC-12）。

### 4.4 CPA / 其他平台扩展（对应 Q4）
- 回调采用适配器注册表：`CodexManagerJsonAdapter`（本期主用）、`HttpWebhookAdapter`、`CsvExportAdapter` 等。
- 新增 CPA 平台 = 新增一个适配器实现 `send(record)`，将统一账号记录映射为目标格式，不改核心。

## 5. 邮箱契约：Cloudflare Email Routing + Email Workers（替代 IMAP）

### 5.1 方案（对应 Q9 确认）
- 在 Cloudflare 配置 **Catch-all → Email Worker**：所有发往 `*@<域名>` 的邮件触发 Worker。
- Email Worker（`email()` handler）解析 `message`（含 `to`、`from`、`headers`、原始 MIME），提取验证码/链接后，**存入可被本软件读取的中转**：
  - **推荐 5.2 拉取式**：写入 Cloudflare KV / D1，本软件通过 Worker 暴露的 HTTP API 按收件人地址轮询拉取。
  - 备选 5.3 推送式：Worker 直接 HTTP 回调本软件本地端点（需软件起本地 HTTP 监听，注意鉴权与可达性）。

### 5.2 推荐：Worker + KV/D1 + HTTP 拉取 API
- Worker 收到邮件 → 解析 → 以 `to` 地址为键写入 KV/D1（含 code、链接、原文、时间）。
- Worker 另暴露受保护的 HTTP 端点（Bearer Token 鉴权）：
  - `GET /mail?to=<addr>&since=<ts>` → 返回该地址最新邮件/验证码。
- 本软件「邮箱监听模块」按 `to` 地址轮询该端点（超时/间隔/最大次数可配）。
- **优点**：本软件无需开放入站端口，纯出站轮询，单机部署友好。

### 5.3 验证码/链接解析
- 在 Worker 端或本软件端用规则解析（**按站点可配置正则**）：
  - 验证码：如 OpenAI 邮件中的数字码（正则按实际邮件模板配置）。
  - 激活/确认链接：提取 URL，由内置浏览器访问完成验证。
- 必须以 `to`（Catch-all 下分配给该任务的唯一地址）精确定位邮件，避免串号。

### 5.4 邮箱地址分配
- 在 Catch-all 域下为每个任务生成唯一本地名（随机串），作为注册邮箱与邮件定位键。

### 5.5 配置项
Cloudflare 域名、Worker API URL、API Token、轮询参数、各站点验证码/链接解析规则。

## 6. 端到端数据流（更新版，含已验证集成）

```
[启动作业]
  → 分配邮箱地址（Cloudflare Catch-all 唯一本地名）
  → 申请手机号（HeroSMS getNumber, service=dr 可配）
  → 内置 Chromium 打开 ChatGPT 注册页
  → 填邮箱 → 邮箱验证（Worker HTTP 拉取 code/链接）
  → 设密码/资料
  → 人机验证（Turnstile/FunCaptcha → CaptchaSolver 打码平台求解）
  → 手机验证（HeroSMS getStatus 取码回填）
  → 注册完成 → 触发 OAuth 登录 → CDP 抓取 access/id/refresh_token + chatgpt_account_id
  → 组装账号记录（含 token 三件套）
  → 本地加密落库
  → 回调：生成 Codex-Manager 兼容 JSON → 写入导入目录（或 account/import RPC）
  → 释放手机号（setStatus=6/8）、销毁会话
  → [成功]
```

### 7.1 R8 推荐默认（开发方拟定，如需调整请提出）
- **用户名**：如需用户名时，`user_` + 8 位随机小写字母数字；ChatGPT 主要以邮箱为账号，用户名按页面实际需要。
- **密码**：16 位，含大小写字母 + 数字 + 符号，满足 OpenAI 强度要求；每账号独立随机。
- **姓名**：随机英文名（FirstName + LastName 词库）。
- **生日**：随机生成且保证 ≥18 岁。
- **邮箱本地名**：Catch-all 域下 10–12 位随机串，保证唯一。
