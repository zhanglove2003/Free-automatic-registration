## 1. 文档目的与范围

### 1.1 目的
定义 Windows 桌面端自动注册软件的**产品级需求**：要做什么、为谁做、边界在哪里、与哪些外部系统交互、产出何种数据、如何验收。本文档不涉及技术实现细节（见《项目设计规范》），不涉及 UI 界面设计（UI 已由设计团队交付）。

### 1.2 范围
- **覆盖**：账号自动注册的完整业务流程、用户角色与用例、功能边界、外部服务交互契约、账号数据字段规范、验收标准。
- **不覆盖**：界面布局/控件样式/交互视觉（UI 已就绪）、具体代码架构（见设计规范）、目标网站的逐站点适配细节（按站点适配清单单独维护，见待澄清）。

### 1.3 术语表
| 术语 | 含义 |
|------|------|
| 内置浏览器 | 软件内嵌的 Chromium 实例，承载注册页面与自动化脚本 |
| 作业 / Job | UI 中的调度单元，对应一个「注册任务」或一次「批量注册运行」 |
| 注册任务 / Task | 完成单个账号注册的最小工作单元 |
| 接码平台 | smshero，提供临时手机号与短信验证码 |
| 邮箱服务 | Cloudflare Email Routing，Catch-all + Email Workers 接收/读取验证邮件（不用 IMAP） |
| 回调保存 | 注册完成后将账号数据通过可扩展回调写出 |
| codex manager | 账号统一管理平台（下游消费方，集成目标，定义见待澄清） |
| CPA 平台 | 后续一键导入账号的营销/获客平台 |

## 2. 产品概述

### 2.1 一句话定义
一款运行于 Windows 的桌面自动化工具：通过**内置 Chromium 浏览器 + 注入脚本**全自动完成目标网站账号注册，自动对接接码平台获取手机验证码、对接 Cloudflare 邮箱接收邮件验证码/链接，注册成功后通过**可扩展回调**输出结构化账号数据，供 CPA 平台与 codex manager 统一管理。

### 2.2 核心价值
- 全自动：从分配资源到落库，无人值守批量产出可用账号。
- 可观测：任务调度面板实时呈现运行中/排队/已完成/失败，含活动日志与时间轴。
- 可恢复：内置异常重试与状态机，失败可单点重试。
- 可扩展：回调与导出采用适配器模式，便于接入不同下游平台。

### 2.3 运行形态
- 单机桌面应用（Windows 10/11，x64）。
- 主窗口即《任务调度管理面板》UI；自动化在后台浏览器实例中执行，与主窗口分离。

## 3. 用户角色与用例

### 3.1 用户角色
| 角色 | 描述 | 主要诉求 |
|------|------|----------|
| 运营/运维管理员（主角色） | 配置并启动批量注册、监控进度、处理失败、导出账号 | 全自动、稳定、可观测、可重试 |
| 配置维护者 | 维护接码/邮箱/目标站点/回调等凭据与参数 | 凭据安全、配置项清晰 |
| 下游消费系统（系统角色） | CPA 平台 / codex manager，接收账号数据 | 数据规范、字段完整、可幂等导入 |

> 说明：UI 当前为单机单人操作模型，无多用户登录/权限分级（见待澄清 Q12）。

### 3.2 核心用例

**UC-01 配置外部服务凭据**
配置维护者在「设置」页录入 smshero API Key、Cloudflare 邮箱 Worker 端点/密钥、打码平台密钥、目标站点参数、回调地址与密钥；系统校验连通性并加密保存。

**UC-02 启动批量注册作业**
管理员设定本次运行的目标站点、注册数量、并发度等参数，点击侧边栏「启动」按钮，系统创建注册任务队列并开始调度执行。

**UC-03 监控运行状态**
管理员在「仪表盘」查看统计卡（运行中/排队/已完成/失败）、作业列表、调度时间轴与最近活动日志，实时掌握进度。

**UC-04 查看任务日志**
管理员点击作业行「日志」按钮，查看该任务的分步执行日志（含外部调用、截图/快照引用、错误堆栈）。

**UC-05 重试失败任务**
对失败任务点击「重试」，系统按重试策略重新执行（可复用或重新申请手机号/邮箱，见待澄清 Q8）。

**UC-06 账号数据落库与回调**
任务成功后，系统将账号记录写入本地加密数据库，并通过回调适配器推送到已配置的下游（CPA/codex manager）。

**UC-07 导入/导出账号**
管理员将账号数据按指定格式导出（CSV/JSON），或一键推送至 CPA 平台 / codex manager。

### 4.2 端到端注册流程（文字流程）
单个注册任务的标准生命周期：

```
[创建任务] 
   → [分配资源] 申请手机号(smshero) + 生成邮箱地址(Cloudflare Catch-all 域下随机本地名)
   → [打开页面] 内置浏览器导航至目标站点注册页
   → [填表] 注入脚本填充：邮箱 / 用户名 / 密码 / 手机号
   → [提交] 触发注册提交
   → [人机校验?] 若有验证码/滑块 → 处理策略(FR-16)
   → [手机验证?] 若需短信码 → 轮询 smshero 取码 → 回填 → 提交
   → [邮箱验证?] 若需邮件验证 → Email Worker 拉取接口轮询 → 解析验证码/激活链接 → 回填或访问链接
   → [登录校验] (可选)用注册凭据登录一次，确认账号可用、抓取 cookie/token
   → [组装记录] 收集 用户名/密码/邮箱/手机号/cookie/token/指纹/代理 等字段
   → [落库] 写入本地加密数据库
   → [回调] 通过适配器推送下游(CPA / codex manager)，按需重试
   → [释放资源] 标记手机号使用状态(smshero)，关闭浏览器会话
   → [任务成功]
```

任意步骤失败 → 进入重试策略（FR-08）；超过最大重试 → 标记「失败」，记录失败原因，等待人工「重试」或忽略。

### 4.3 状态机
任务状态：`排队(queued)` → `运行中(running)` →（可暂态 `等待短信(wait_sms)` / `等待邮件(wait_email)` / `等待人机校验(wait_captcha)`）→ `成功(success)` 或 `失败(failed)`。
- `failed` 可经「重试」回到 `queued`。
- UI 统计卡映射：运行中=running、排队=queued、已完成=success、失败=failed。

### 4.4 UI 事件 → 后台逻辑映射
> 仅说明界面事件触发的后台行为，不涉及界面设计。

| UI 事件（来自交付的面板） | 触发的后台逻辑 |
|---------------------------|----------------|
| 侧边栏「启动」按钮 | 读取当前运行配置，创建任务队列，启动调度器 |
| 「仪表盘」统计卡/作业列表 | 订阅任务状态变更，实时刷新计数与行状态 |
| 作业行「日志」按钮 | 拉取该任务分步日志并展示 |
| 作业行「重试」按钮 | 以原参数重新入队该任务 |
| 「设置」页保存 | 校验并加密保存外部服务凭据与参数 |
| 「日志」导航页 | 展示全局运行日志（按时间/级别/任务过滤） |
| 主题切换 | 仅前端视觉，无后台逻辑 |

## 5. 功能边界（做什么 / 不做什么）

### 5.1 做什么（In Scope）
- 在内置 Chromium 中自动完成目标网站账号注册全流程。
- 对接 smshero 获取手机号与短信验证码。
- 对接 Cloudflare Email Routing（Catch-all 转发 + Email Worker）接收并解析验证邮件。
- 任务调度、并发控制、异常重试、分步日志、状态监控。
- 账号数据本地加密存储与可扩展回调推送。
- 按字段规范导出账号数据。

### 5.2 不做什么（Out of Scope，本期）
- 不提供 UI 设计/改版（UI 已交付，直接使用）。
- 不做服务端/云端部署，纯单机桌面应用。
- 不内置自研破解强人机验证的算法；已确认通过**对接第三方打码平台**处理 Turnstile/FunCaptcha（平台与密钥部署时配置）。
- 不实现 codex manager / CPA 平台本身，仅做对接（按其接口契约，见待澄清 Q3、Q4）。
- 不做多租户/多用户权限体系（见待澄清 Q12）。
- 不保证对任意网站的通用适配；仅支持「站点适配清单」内站点（见待澄清 Q2）。
- 暂不接入除 Cloudflare 外的其他邮箱服务（本期明确仅 Cloudflare）。

### 5.3 合规与使用约束（重要）
- 自动批量注册可能触及目标网站服务条款与当地法规。本软件定位为**授权范围内**的营销/获客自动化工具，使用方须确保对目标站点具备合法授权与合规依据。
- 文档不提供绕过安全机制、规避风控检测以实施滥用的实现细节；相关能力仅在合法授权前提下使用。

## 6. 与外部服务的交互定义

> 以下为**业务契约层**定义（输入/输出/触发时机/失败语义）；具体 API 端点、鉴权头、字段映射见《项目设计规范》第「接口设计」章，未知细节列入待澄清。

### 6.1 smshero（接码平台）
| 交互 | 触发时机 | 输入 | 期望输出 | 失败语义 |
|------|----------|------|----------|----------|
| 申请手机号 | 任务进入「分配资源」 | 国家/服务/项目标识 | 手机号 + 订单/会话 ID | 无号可用→延迟重试或任务失败 |
| 轮询验证码 | 提交手机号后 | 订单 ID | 短信验证码 | 超时未到→重试/换号策略(Q8) |
| 完成/释放号码 | 取码成功或任务结束 | 订单 ID + 状态 | 确认 | 失败仅告警，不阻断 |
| 标记号码无效 | 收不到码/号码被拒 | 订单 ID | 确认 | 触发换号 |

要求：所有调用遵循 smshero 官方 HTTP API 规范；需可配置超时、最大轮询次数、轮询间隔；费用相关行为（拉黑/退款）按平台规则处理（参数见 Q7）。

### 6.2 Cloudflare Email Routing（邮箱服务）
> 已确认（Q9）：采用 **Cloudflare Email Routing + Email Workers/API** 接收与读取邮件，**不使用 IMAP**。实现细节见《03 集成对接契约》邮箱章。

| 交互 | 说明 |
|------|------|
| 邮箱地址生成 | 在已配置的 Catch-all 域名下生成随机本地名（如 `u8f3k2@yourdomain.com`），作为注册邮箱 |
| 邮件接收 | Cloudflare Email Routing 将该域邮件交给绑定的 **Email Worker** 处理（而非转发到第三方收件箱） |
| 邮件读取 | Email Worker 解析邮件后，将结构化结果（收件人、主题、正文、验证码候选）写入 **Workers KV / D1**，或经 HTTP 推送/拉取接口供注册机查询 |
| 验证码解析 | 在 Worker 内或客户端用正则/规则从正文（text/html）提取数字验证码 |
| 激活链接解析 | 提取激活/确认 URL，由内置浏览器访问完成验证 |

要求：
- 必须支持按「收件人地址精确匹配」定位某任务邮件（Catch-all 汇入，需用 To/收件人区分）。
- 支持轮询/拉取（超时、间隔、最大次数可配）。
- 解析规则按站点可配置（每站点验证码正则 / 链接特征）。

### 6.3 目标网站（已确认：OpenAI / ChatGPT）
> 已确认（Q2）：首批目标为 **ChatGPT（OpenAI）账号注册**。

- 目标产出不仅是「用户名/密码」，而是**完整可用的 OpenAI 账号 + OAuth 登录态**（`access_token` / `id_token` / `refresh_token` 等），因为下游 Codex-Manager 以 OAuth token 包导入（见 6.5 与《03 集成对接契约》）。
- 注册流程须包含：邮箱注册 → 邮箱验证 → （如需）手机验证 → **完成 OAuth 登录并抓取 token**。
- ChatGPT 注册含强人机校验（如 Cloudflare Turnstile / Arkose FunCaptcha），处理策略见 6.x 与待澄清 Q5（策略未定，影响成功率）。
- 站点适配描述（注册页 URL、字段选择器、OAuth 回调抓取点、成功判定）见《02 设计规范》与《03 集成对接契约》。

### 6.4 回调保存（下游推送）
- 任务成功后，按配置将账号记录推送到一个或多个下游目标。
- 采用**适配器模式**：每个下游一个适配器（HTTP Webhook / 文件 / 数据库 / 平台专用 API）。
- 默认提供 HTTP Webhook 适配器：POST JSON，带签名（HMAC）与幂等键。
- 失败具备重试队列；多次失败转入「待重投」并告警。

### 6.5 codex manager（统一管理）— 已确认
> 已确认（Q3）：下游为开源项目 **Codex-Manager**（`github.com/qxcnm/Codex-Manager`，Tauri + Rust，Codex CLI 账号管理与本地网关转发）。

- 接入方式（经源码核实，见《03 集成对接契约》）：Codex-Manager 以 **JSON token 包**导入账号，支持三种入口：①目录导入（扫描 `.json`）②文件导入 ③`account/import` RPC。
- **字段契约（必须满足）**：每个账号一个 JSON，至少含 `access_token`；推荐含 `id_token`、`refresh_token`、`chatgpt_account_id`，可选 `meta`（label/group_name/note/tags/workspace_id）、顶层 `email`（作账号标签）。
- 本注册机的回调/导出**必须产出与该 JSON 结构兼容的文件**，使账号可被 Codex-Manager 直接导入。

### 6.6 CPA 平台
- 通过导出文件或专用回调适配器一键导入账号。
- 具体格式与字段映射待提供（见待澄清 Q4）。

## 7. 数据字段规范（账号记录）

每条成功账号记录至少包含下列字段。标注「加密」者须以密文落盘（见设计规范安全章）。

| 字段 | 类型 | 必填 | 说明 | 敏感 |
|------|------|------|------|------|
| id | string(UUID) | 是 | 记录唯一 ID | 否 |
| site | string | 是 | 目标站点标识 | 否 |
| username | string | 是 | 注册用户名/账号名 | 否 |
| password | string | 是 | 账号密码 | 加密 |
| email | string | 是 | 注册邮箱（Cloudflare 域地址） | 否 |
| email_inbox | string | 否 | 实际接收邮件的真实收件箱 | 否 |
| phone | string | 否 | 注册手机号（如使用） | 否 |
| phone_country | string | 否 | 手机号国家/区号 | 否 |
| sms_order_id | string | 否 | smshero 订单/会话 ID | 否 |
| cookies | json | 否 | 登录后 cookie 集合 | 加密 |
| token | string | 否 | 访问/会话 token（兼容旧字段） | 加密 |
| oauth_access_token | string | 是(ChatGPT) | OpenAI OAuth access_token，Codex-Manager 导入必填 | 加密 |
| oauth_id_token | string | 否 | OpenAI OAuth id_token（含 chatgpt_account_id/workspace_id 声明） | 加密 |
| oauth_refresh_token | string | 否 | OpenAI OAuth refresh_token，供长期刷新 | 加密 |
| chatgpt_account_id | string | 否 | ChatGPT 账号 ID（Codex-Manager 去重/分组用） | 否 |
| workspace_id | string | 否 | OpenAI workspace ID（如有） | 否 |
| account_status | enum | 是 | created / verified / login_ok / unusable | 否 |
| proxy | string | 否 | 本次使用的代理（如有） | 加密 |
| fingerprint | json | 否 | 浏览器指纹快照（如有） | 否 |
| extra | json | 否 | 站点特有附加字段（可扩展） | 视内容 |
| task_id | string | 是 | 关联注册任务 ID | 否 |
| created_at | datetime | 是 | 创建时间(UTC) | 否 |
| verified_at | datetime | 否 | 验证完成时间 | 否 |
| callback_status | enum | 是 | pending / sent / failed | 否 |
| callback_attempts | int | 是 | 回调尝试次数 | 否 |

> 字段可按站点适配扩展，扩展字段统一放入 `extra`，不破坏核心结构。
> **导出/回调到 Codex-Manager 时**，须将上述 `oauth_*` 字段映射为其导入 JSON 的 `access_token`/`id_token`/`refresh_token`/`chatgpt_account_id`（映射表见《03 集成对接契约》）。CPA 平台字段映射见待澄清 Q4。

## 8. 验收标准

### 8.1 功能验收
- **AC-01**：在「设置」页正确录入凭据后，系统可分别校验 smshero、Cloudflare Email Worker、打码平台、回调地址连通性并给出明确结果。
- **AC-02**：点击「启动」可创建任务队列并按最大并发执行，仪表盘统计与作业列表实时反映状态。
- **AC-03**：对一个「需手机验证」站点，系统能自动申请号码、取码、回填并通过验证。
- **AC-04**：对一个「需邮件验证」站点，系统能通过 Catch-all + Email Worker 取到对应邮件、解析验证码或访问激活链接并通过验证。
- **AC-05**：注册成功的账号记录字段完整（第 7 章），敏感字段在数据库中为密文。
- **AC-06**：成功任务触发回调，下游收到带签名与幂等键的 JSON；回调失败可自动重试。
- **AC-07**：任意步骤异常可按策略重试；超限标记失败并记录原因；「重试」按钮可重新执行。
- **AC-08**：每个任务有可查询的分步日志，含外部调用结果与错误详情。
- **AC-09**：可按字段规范导出 CSV/JSON。

### 8.2 质量与非功能验收
- **AC-10**：在配置的最大并发下连续运行 ≥100 个任务不崩溃、无内存泄漏导致的持续增长。
- **AC-11**：批量运行可稳定产出账号（成功率阈值本期不作硬性要求，Q10 暂不要求；受打码平台与无代理策略影响，以实跑数据为准）。
- **AC-12**：凭据与敏感字段不以明文出现在数据库、日志或导出（除非显式导出敏感字段且经确认）。
- **AC-13**：异常关闭后重启，未完成任务状态可恢复或安全标记，不丢失已成功账号数据。
- **AC-14**：单任务端到端耗时在合理范围（本期不设硬性上限，Q10 暂不要求；含打码与等待短信/邮件，以实跑数据为准）。

### 8.3 可观测性验收
- **AC-15**：仪表盘统计卡、作业列表、时间轴、活动日志数据来自真实任务运行，非静态示例。
- **AC-16**：日志可按时间、级别、任务 ID 过滤检索。
